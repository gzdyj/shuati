## 一、问题分析

### 1.1 乱码问题现状

根据V2/V3文档记录，虽然做了字符编码配置，但前端仍显示乱码。需要排查：

| 排查项 | 当前状态 | 排查方向 |
|--------|---------|---------|
| 数据库连接URL | characterEncoding=utf8 | 需要确认是否生效 |
| 前端数据源 | 首页使用硬编码分类数据 | 需要改为从API获取 |
| API响应编码 | 已配置UTF-8 | 需要验证实际响应头 |
| 数据库表字符集 | utf8mb4 | 需要确认实际字符集 |

### 1.2 Excel导入问题

| 问题描述 | 根因分析 | 解决方案 |
|---------|---------|---------|
| 1万条数据导入失败 | 逐条插入效率低，事务超时 | 改为批量插入，分批提交 |
| 大文件解析慢 | POI内存消耗大 | 使用SXSSFWorkbook流式解析 |
| 上传超时 | 文件传输时间长 | 检查文件大小限制配置 |

### 1.3 脏数据问题

| 数据类型 | 问题描述 | 处理方式 |
|---------|---------|---------|
| 分类数据 | 乱码或重复 | 清理后重新初始化 |
| 题目数据 | 乱码或不完整 | 清理后重新导入 |
| 选项数据 | 与题目不匹配 | 级联删除 |

### 3.3 脏数据清理方案

**清理脚本**:
```sql
-- 删除无关联的选项
DELETE FROM question_option WHERE question_id NOT IN (SELECT id FROM question);

-- 删除内容为空的题目
DELETE FROM question WHERE content IS NULL OR content = '';

-- 删除乱码分类
DELETE FROM category WHERE name REGEXP '[\\x00-\\x1F\\x7F]';
```

## 六、踩坑预防清单

### 6.1 历史踩坑经验（V1/V2/V3）

| 问题类型 | 问题描述 | 解决方案 |
|---------|---------|---------|
| 字符编码 | MySQL字符集不一致导致乱码 | 统一使用utf8mb4，配置characterSetResults |
| PowerShell | &&语法错误 | 使用分号分隔命令 |
| Docker | MySQL启动慢导致后端连接失败 | 使用healthcheck等待 |
| 端口冲突 | 3306/8080端口被占用 | 修改映射端口 |
| Excel解析 | POI列索引从0开始 | 注意Excel第B列对应索引1 |
| 事务处理 | 批量操作需要事务管理 | 添加@Transactional(rollbackFor = Exception.class) |

### 6.2 V4潜在风险

| 风险点 | 预防措施 |
|--------|---------|
| 大文件内存溢出 | 使用SXSSFWorkbook流式解析 |
| 批量插入超时 | 分批提交，每批500条 |
| 脏数据清理不完整 | 清理前备份数据 |
| 导入重复数据 | 添加去重逻辑 |
